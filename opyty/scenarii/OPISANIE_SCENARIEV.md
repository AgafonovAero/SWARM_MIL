# Описание сценариев

## Назначение сценариев

Сценарии задают проверяемый состав исходных данных для будущих опытов с роем
БВС. На этапе 2 они описывают только структуру данных, начальные условия,
ограничения и ожидаемые показатели. Сценарии не выполняют моделирование,
не запускают движение и не содержат реализацию связи, управления,
обучения или блочных моделей.

## Обязательные поля сценария

Каждый сценарий обязан содержать поля:

- `id_scenariya`;
- `nazvanie`;
- `opisanie`;
- `nachalnoe_chislo`;
- `vremya_modelirovaniya`;
- `shag_modelirovaniya`;
- `oblast_poleta`;
- `sostav_roya`;
- `tseli_zadaniya`;
- `prepyatstviya`;
- `zony_zapreta`;
- `otkazy`;
- `pomekhi_svyazi`;
- `ogranicheniya_bezopasnosti`;
- `ozhidaemye_pokazateli`.

Поле `ogranicheniya_bezopasnosti` на этапе 2 должно содержать защитные
признаки:

- `issledovatelskii_harakter`;
- `net_realnogo_upravleniya`;
- `net_vrednogo_naznacheniya`.

## Правила задания состава роя

- состав роя задается массивом записей о БВС;
- каждый БВС имеет уникальный идентификатор;
- для каждого БВС задаются роль, начальное положение, начальная скорость
  и запас энергии;
- начальные скорости на этапе 2 используются только как исходные данные;
- состав роя не описывает правила движения, управления и взаимодействия.

## Правила задания области полета

- область полета задается прямоугольными границами по трем осям;
- для каждой оси задаются нижняя и верхняя граница;
- верхняя граница должна быть строго больше нижней;
- все начальные положения БВС, препятствия и зоны запрета должны
  находиться внутри области полета.

## Правила задания препятствий

- препятствия описываются только геометрически;
- для препятствия задается идентификатор, тип и границы;
- на этапе 2 препятствия не влияют на движение, а только проверяются
  на корректность размещения внутри области полета;
- не допускается описание способов преодоления или обхода препятствий.

## Правила задания зон запрета

- зона запрета описывается как геометрическая область внутри общей области
  полета;
- каждая зона запрета имеет идентификатор, название и границы;
- зона запрета на этапе 2 служит только для проверки исходных данных;
- поведение роя внутри зоны запрета на этапе 2 не моделируется.

## Правила задания целей задания

- цели задания описываются в исследовательской и учебной форме;
- формулировки целей не должны содержать поражение, наведение на людей
  или причинение вреда;
- для каждой цели задаются идентификатор, тип, описание и критерий
  будущей оценки;
- цель задания определяет, для чего сценарий будет использоваться
  на следующих этапах.

## Правила задания защитных признаков сценария

- защитные признаки задаются только в поле
  `ogranicheniya_bezopasnosti`;
- признак `issledovatelskii_harakter` фиксирует исследовательский и
  учебный характер постановки;
- признак `net_realnogo_upravleniya` подтверждает, что сценарий не
  предназначен для управления реальными БВС;
- признак `net_vrednogo_naznacheniya` подтверждает, что сценарий не
  описывает причинение вреда и не задает вредное назначение;
- все защитные признаки на этапе 2 должны быть равны `true`.

## Правила задания отказов

- отказ задается как будущее событие сценария с указанием времени;
- отказ ссылается только на существующий БВС;
- отказ на этапе 2 описывает потерю работоспособности участника в модели
  без расчета последствий;
- не допускается описание воздействия на реальные объекты.

## Правила задания помех связи

- помеха связи задается как будущий расчетный фактор ухудшения качества
  передачи сведений;
- каждая помеха имеет идентификатор, временные границы, область действия
  и коэффициент ухудшения;
- на этапе 2 не допускается описание реального подавления связи,
  средств радиоэлектронного воздействия или практического применения;
- помеха используется только как часть проверяемого состава данных.

## Правила фиксации начального числа случайных величин

- начальное число случайных величин задается целым числом;
- одно и то же начальное число должно обеспечивать повторяемость
  последующего расчета;
- начальное число фиксируется в каждом сценарии отдельно;
- изменение начального числа считается изменением исходных данных сценария.

## Правила именования сценариев

- имя файла сценария должно быть кратким и устойчивым для сопровождения;
- идентификатор сценария должен быть уникальным;
- название сценария и описание сценария пишутся по-русски;
- имя файла и идентификатор выбираются так, чтобы их было удобно
  использовать в MATLAB, Git и GitHub.

## Ограничения этапа 2

- сценарии задают только исходные данные и не выполняют моделирование;
- сценарии являются исходными данными опыта, а не расчетной моделью;
- кинематика и движение БВС на этапе 2 не реализуются;
- связь, управление, обучение и деление роя на звенья не реализуются;
- модели Simulink не создаются;
- внешние наборы данных не используются;
- сценарии сохраняют исследовательский, учебный и модельный характер.
